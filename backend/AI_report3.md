## 整体目标

基于 **FastAPI + LangChain + FAISS + Sentence-Transformers + Vue3/TS**，实现一个多智能体旅行助手，包括：  
景点搜索专家、酒店推荐专家、天气查询专家、行程规划专家（统筹/协调者）。下面是按阶段拆解的落地路线图。

---

## 第 1 阶段：需求与架构梳理

- **明确功能边界与数据来源**
  - **景点/酒店**：选定 1–2 个真实 API（如某地图/旅游平台开放接口）或自建小型 Demo 数据库，确保“必须用工具查询，不能编造”有落地实现。
  - **天气**：选定天气 API（如 OpenWeather / 和风天气等）。
- **系统架构草图**
  - **后端**：`FastAPI` + `LangChain`（多 Agent & 工具调用）+ `FAISS`（本地向量库）。
  - **模型**：优先选一个支持函数/工具调用的 Chat 模型；Sentence-Transformers 选中文/多语种模型（如 `paraphrase-multilingual-MiniLM` 系列）。
  - **前端**：`Vue 3 + TypeScript + Vite`，REST/WS 与后端交互。

---

## 第 2 阶段：基础工程搭建

- **后端项目骨架**
  - **结构示例**：
    - `app/main.py`（FastAPI 入口）
    - `app/config.py`（配置/环境变量）
    - `app/models/`（Pydantic 模型）
    - `app/tools/`（各类 LangChain 工具）
    - `app/agents/`（各个智能体与调度逻辑）
    - `app/vectorstore/`（FAISS + Embedding 管理）
    - `app/api/`（路由：行程规划、健康检查等）
  - 引入依赖：`fastapi`, `uvicorn`, `langchain`, `sentence-transformers`, `faiss-cpu`, `httpx/requests`, `pydantic`, 日志库等。
- **前端项目初始化**
  - 使用 `create-vue` 或 Vite 手动初始化：`Vue3 + TS + Vue Router + Pinia`。
  - 规划基础路由：`/`（主页+输入偏好）、`/plan`（查看行程结果）。

---

## 第 3 阶段：向量库与知识底座

- **选择与加载 Sentence-Transformers 模型**
  - 在后端初始化：只加载一次的全局 `SentenceTransformer` 实例（注意异步/线程安全）。
- **构建 FAISS 向量库**
  - **数据内容**：常见城市简介、热门景点简要说明、交通/签证/安全等通用旅行知识 —— 可先用静态 Markdown/JSON 构造少量样例。
  - 写入流程：
    - 清洗文本 → 切分成 `Document` → 使用 `SentenceTransformerEmbeddings` 生成向量 → 存入 `FAISS`。
  - **持久化**：将向量库保存到本地磁盘，提供加载函数（服务重启后快速恢复）。
- **封装检索工具**
  - 用 LangChain 提供 `VectorStoreRetriever`，包装成 `Tool`（如 `kb_retriever_tool`），供多个智能体共享做“背景知识补全”。

---

## 第 4 阶段：工具层设计（必须用工具、不编造）

- **统一 HTTP 客户端 & 错误处理**
  - 封装调用外部 API 的 `HttpClient`（使用 `httpx`/`requests`），支持：
    - 超时与重试
    - 统一异常 → 结构化错误信息返回给 LLM，避免它胡编。
- **景点搜索工具**
  - `search_attractions_tool`：
    - 入参：城市/目的地、日期区间（可选）、偏好标签（人文/美食/亲子/自然等）、预算等级等。
    - 实现：调用真实旅游/地图 API 或查询自建景点表。
    - 返回：结构化列表（名称、简介、评分、经纬度、标签、开放时间、价格）。
- **酒店搜索工具**
  - `search_hotels_tool`：
    - 入参：城市/区域、入住/离店日期、价位区间、偏好（交通便利/干净安静/亲子友好等）。
    - 实现：对接酒店 API 或自建酒店数据。
    - 返回：名称、位置（坐标+地址）、价格、评分、设施标签等。
- **天气查询工具**
  - `weather_query_tool`：
    - 入参：城市/经纬度、日期范围。
    - 实现：调用天气 API。
    - 返回：每天的天气概况（温度范围、降水概率、天气描述、极端天气预警）。
- **辅助工具**
  - 可选：距离计算/地图工具（根据经纬度计算景点与酒店的距离），为行程规划提供“是否就近合理”的依据。

## 第 5 阶段：四个智能体设计（LangChain 层）

### 5.1 景点搜索专家 Agent

- **职责**：只负责“找到/筛选/对比景点”，不生成完整行程。
- **工具配置**：
  - 核心工具：`search_attractions_tool`、`kb_retriever_tool`（补充描述，不作为事实来源）。
- **系统提示词要点**：
  - 必须通过工具获取景点信息，不得凭空编造名称/价格/评价。
  - 遇到工具返回为空时，要明确告诉用户“没有找到符合条件的景点”而不是臆造。
  - 输出格式：结构化 JSON（如按天/类别分组），方便后续统筹 Agent 使用。
- **实现方式**：
  - 使用 LangChain 的工具调用 Agent（如基于 `ChatOpenAI` + 工具），对接 FastAPI 依赖注入。

### 5.2 酒店推荐专家 Agent

- **职责**：依据行程大致区域和用户偏好，给出多个备选酒店。
- **工具配置**：
  - 核心工具：`search_hotels_tool`，可选使用距离工具辅助“离景点是否方便”。
- **系统提示词要点**：
  - 优先考虑靠近多数景点/起点交通枢纽。
  - 需说明推荐理由（位置、性价比、适配目标人群等）。
  - 仍然必须仅依据工具返回数据，不得自造酒店信息。
- **输出格式**：按城市/日期段返回 3–5 个候选酒店列表，包含关键字段。

### 5.3 天气查询专家 Agent

- **职责**：针对整个行程日期区间，给出按天的天气情况与风险提示。
- **工具配置**：
  - 核心工具：`weather_query_tool`。
- **系统提示词要点**：
  - 不预测超过 API 提供范围的日期；超出部分要明确“不确定，建议临近出发再查”。
  - 根据天气给出简要建议（如雨天备伞、炎热时段室内活动等）。
- **输出格式**：按日期的天气 JSON，含建议字段。

### 5.4 行程规划专家 Agent（统筹者）

- **职责**：整合前三个专家的结果 + 用户偏好，输出完整行程计划。
- **输入信息**：
  - 用户原始需求（出发地、目的地、天数、预算、节奏偏好等）。
  - 景点专家返回结果。
  - 酒店专家返回结果。
  - 天气专家返回结果。
- **系统提示词要点**：
  - 以“减少来回折腾、每天行程合理”为目标。
  - 规避明显不合理安排（如户外活动排在暴雨天）。
  - 明确标记每一天的：早/中/晚安排、对应景点/餐饮/酒店、交通说明。
- **实现方式（协调机制）**：
  - v1：用 Python 函数作为 orchestrator：
    - Step1：调用景点 Agent → 得到候选景点。
    - Step2：调用酒店 Agent → 得到候选酒店。
    - Step3：调用天气 Agent → 获取天气。
    - Step4：将上述结果连同用户偏好打包，传给行程规划 Agent（单次调用 LLM）。
  - v2（扩展）：引入更复杂的多 Agent 框架（如基于 LangChain 的多 Agent 或 LangGraph）做循环优化/协商。

## 第 6 阶段：工作流与数据模型设计

- **Pydantic 数据模型**
  - **请求体**：`TripRequest`（目的地、出发日期、天数、预算、偏好标签、出行人信息等）。
  - **中间结果**：`AttractionCandidate`, `HotelCandidate`, `DailyWeather`。
  - **最终结果**：`DayPlan`, `TripPlan`（包含每天的活动安排、住宿、天气摘要、备注）。
- **工作流编排**
  - 使用 LangChain 的 `RunnableSequence` 或普通 async 函数串接：
    - 请求解析/归一化 → 调用专家 Agents（可并行） → 收集结果 → 交给行程规划专家 → 返回标准化 `TripPlan`。
  - 为每一步增加日志（输入/输出摘要）方便调试和评估。

---

## 第 7 阶段：FastAPI 接口设计

- **核心接口**
  - `POST /plan-trip`
    - 请求体：`TripRequest`
    - 响应体：`TripPlan`
    - 内部：调用 orchestrator 执行多智能体流程。
  - `GET /healthz`
    - 返回服务状态、外部 API 探活结果（可选）。
- **可选功能**
  - **流式返回**：使用 `StreamingResponse` 或 WebSocket，将“分析中 / 已完成景点候选 / 已生成行程草案 / 最终行程”分阶段发送给前端。
  - **用户会话 ID**：允许前端持久化一次会话，多轮微调同一行程。

---

## 第 8 阶段：前端（Vue3 + TS）界面实现

- **信息结构与交互**
  - **输入区**：表单或聊天式输入：
    - 目的地、日期、天数、预算、出行人、兴趣标签等。
  - **结果展示区**：
    - 按天时间轴展示行程（早/中/晚模块卡片）。
    - 每个景点/酒店卡片展示名称、评分、距离（若有）、预算等级。
    - 天气信息以图标+温度/降水提示的形式附在日期标题上。
- **技术实现**
  - **API 层**：使用 `axios` 或 `fetch` 封装调用 `POST /plan-trip`。
  - **状态管理**：用 `Pinia` 管理当前请求状态、计划结果、加载/错误状态。
  - **组件拆分**：
    - `TripForm`（偏好输入）
    - `TripTimeline`（行程时间轴）
    - `DayPlanCard`（单日详情）
    - `AttractionCard` / `HotelCard` / `WeatherBadge`
- **用户体验优化**
  - 提交后 Skeleton Loading / 进度提示（“正在搜索景点…/正在规划行程…”）。
  - 错误提示清晰（如外部 API 限流、信息不足等）。

---

## 第 9 阶段：日志、监控与评估

- **调用链路记录**
  - 记录每次请求的：
    - 用户需求摘要
    - 每个 Agent 的工具调用（参数和简要结果）
    - 最终行程概要
  - 可选接入 LangSmith 或自建简单日志查看页面。
- **效果评估**
  - 准备固定测试用例（如“3 天上海亲子游”、“5 天东京自由行”等）。
  - 人工检查：景点/酒店是否真实存在、安排是否合理、天气是否正确引用。
  - 根据评估结果反复调优：
    - Prompt（系统提示词）
    - 工具参数策略（搜索半径、价格区间等）
    - 行程规划策略（节奏、分配）。

---

## 第 10 阶段：部署与迭代

- **部署**
  - 后端：容器化（Docker），使用 `uvicorn` + 反向代理（如 Nginx / Caddy）。
  - 前端：构建静态资源，部署到任意静态站点服务（如 Nginx/OSS/CDN）。
  - 使用环境变量管理各类 API Key、模型配置等。
- **后续迭代方向**
  - 多城市/跨城市行程（自动切换城市的景点/酒店/天气）。
  - 支持多轮对话微调：用户“删掉这个景点”“改成慢节奏”等再规划。
  - 增加预算估算与费用拆分、团队成员偏好合并等高级能力。
