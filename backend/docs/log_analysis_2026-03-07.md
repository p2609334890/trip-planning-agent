- **23:28:30** 子专家完成（景点 7974 字、天气 37 字、酒店 1699 字）。
- **23:33:30** 出现 `Retrying request to /chat/completions`（openai._base_client），说明规划 Agent 调 LLM 时失败（超时或 5xx），触发重试。
- **23:37:00** 同一 Task-71 才 HTTP 200。

**影响**：用户要等约 4 分钟才能拿到规划结果，体验差。

**建议**：
- 适当调大 LLM 客户端 timeout，或对规划阶段使用更长超时。
- 规划 Agent 的 prompt 或输入过长（如 7974+37+1699 字）可能导致模型响应慢，可考虑对子专家结果做摘要或截断后再传给规划 Agent。
- 对 `/chat/completions` 增加重试上限与退避，并打 ERROR 日志便于排查。

### 2. 景点搜索被拆成多轮，请求翻倍

时间线：

- **23:26:09** 景点搜索 1 次：`city=邢台, preferences=历史,美食` → 高德 keywords=历史,美食，返回 **1 个**。
- **23:26:29** 景点搜索 2 次：`preferences=历史`、`preferences=美食`（各 12 max_results），返回 4 + 12 个。
- **23:26:50** 景点搜索再 2 次：`preferences=古迹`、`preferences=风景`，各返回 12 个。

说明景点子专家内部多次调用了 `search_attractions`，把「历史,美食」拆成多组关键词，甚至扩展成「古迹」「风景」。

**影响**：高德调用次数多、子专家耗时增加（约 2 分钟）、传给规划 Agent 的景点文本过大（7974 字），加剧规划阶段超时风险。

**建议**：
- 在景点专家 system prompt 中约束：一次请求只调用一次 `search_attractions`，关键词用用户偏好的综合描述（如「历史,美食」），不要拆成多次。
- 或在编排层只触发一次景点专家，传入合并后的 preferences，避免由模型自发拆词多轮调用。

### 3. 邢台「历史,美食」首次只返回 1 个景点

- 高德 POI：`city=邢台, keywords=历史,美食` 只返回 **1 个**。
- 后续用「历史」「美食」分开搜才分别得到 4 和 12 个。

**可能原因**：高德对「历史,美食」整体匹配较严，或邢台 POI 分类/数据较少。

**建议**：
- 后端对 `preferences` 做简单拆分（如按逗号），用多关键词分别请求高德后**去重合并**，既保证数量又避免子专家多次 tool call。
- 或保留一次「综合关键词」请求，再补一次「拆分关键词」请求，合并去重后返回。
