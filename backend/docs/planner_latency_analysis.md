# 规划耗时代码与日志分析

基于修改后的日志（多关键词合并、单处「子专家完成」日志）与当前代码，分析**用时长**的原因。

---

|------------|------|------|

---

## 二、用时长原因（结合代码）

### 1. 规划阶段单次 LLM 超时（主因）

**现象**：`LLM_TIMEOUT=300`（5 分钟），规划 Agent 第一次调用在约 5 分钟后触发 `Retrying request`，说明**单次请求在 300s 内未返回**。

**代码依据**：

- `app/config.py` / `.env`：`LLM_TIMEOUT=300`。
- `specialized_agents.py` 中 `_get_llm()` 对所有 Agent 共用同一 timeout：

```python
def _get_llm():
    return ChatOpenAI(
        ...
        timeout=settings.LLM_TIMEOUT,  # 300
    )
```

- 规划 Agent 与子专家共用该 LLM（`_create_planner_agent()` 也调 `_get_llm()`）。

**原因**：规划 Agent 的 system prompt（`PLANNER_AGENT_PROMPT`）很长（约 3k+ 字符），且要求模型输出**整份多日行程 JSON**（含每日景点、餐饮、酒店、天气、预算等），输出 token 数大、生成时间长，下游接口在 300s 内未返回即被客户端判定超时并重试。

**结论**：规划阶段一次成功就需要约 **4–6 分钟**，当前 300s 超时偏紧，导致「第一次超时 + 重试再等约 4 分钟」，总时长约 10 分钟。

---

### 2. 子专家阶段约 90 秒

- 00:15:25 → 00:16:55 约 **90 秒**。
- 景点搜索：多关键词、合并去重已在 1 秒内完成（00:15:26 日志）。
- 三个子专家由 `asyncio.gather` **并行**执行（`plan_trip_async` 543–558 行），所以子专家阶段耗时 ≈ **最慢的那条链**。
- 每条链：一次 LLM 决策 → 调 tool（高德/天气/酒店）→ 再一次 LLM 汇总。三次 HTTP 200 分别约 33s、53s、3s，整体被最慢的一条（约 53s）拖住，加上工具调用和调度，总约 90s 合理。

**结论**：子专家并行已生效，多关键词合并也减少了景点侧重复调用；当前约 90s 主要来自「每条链的 LLM+工具」本身，不是重复请求导致。

---

### 3. 规划阶段无单独超时与耗时日志

- 规划 Agent 与子专家共用 `_get_llm()`，无法为规划单独放宽超时。
- 没有「规划开始/结束」的耗时日志，难以在日志里直接看到规划阶段占用的时间。

---

## 三、建议改动（代码层面）

1. **规划阶段使用更长超时**  
   - 为规划 Agent 单独建一个 LLM 实例（或单独 timeout 配置），例如 **600s**，避免首次正常但较慢的响应被 300s 截断并触发重试。
2. **打规划阶段耗时日志**  
   - 在 `plan_trip_async` 中，在调用规划 Agent 前后打日志（或计时），记录「规划开始」「规划完成，耗时 X 秒」，便于后续对比和调优。
3. **可选：压缩规划输入**  
   - 若后续仍希望控制在 300s 内，可对传入规划 Agent 的「景点/酒店/天气」做摘要或截断（例如最多 N 条景点、酒店），减少输入与输出规模，从而缩短生成时间。

---

## 四、小结

| 阶段       | 耗时（约） | 原因 |
|------------|------------|------|
| 子专家     | ~90s       | 并行 3 条链，最慢一条 LLM+工具 约 53s，总体合理。 |
| 规划首次   | 300s 后超时 | 输出 JSON 过大，单次生成超过 300s，触发重试。 |
| 规划重试   | ~222s      | 重试后成功，说明单次成功约需 4–6 分钟。 |
| **合计**   | **~10 分钟** | 主要来自规划阶段 300s 超时 + 重试约 4 分钟。 |

**根本原因**：规划 Agent 单次 LLM 调用所需时间**超过当前 300s 超时**，导致一次超时重试，总时长翻倍。  
**推荐**：为规划阶段单独设置更长超时（如 600s），并加上规划阶段耗时日志，便于验证和继续优化。
